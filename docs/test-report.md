# ohmcp 测试方案与测试报告

## 1. 测试目标

1. 功能正确性：帧编解码、压缩、加解密、认证、ACL、缓存、端到端调用语义；
2. 性能量化：在与官方 MCP SDK 同传输语义的 JSON-RPC 基线对比下，验证
   通信效率（线上字节）、吞吐、时延的提升幅度（赛题要求 ≥ 10%）；
3. 安全性：令牌不过网、密文防篡改、未认证拒绝服务、越权工具调用被拒。

## 2. 测试环境

| 项 | 值 |
|---|---|
| OS | Linux 5.15 x86_64 |
| CPU | 2 vCPU |
| Rust | rustc 1.83.0，release + `target-cpu=native` |
| 传输 | 同机 Unix Domain Socket |
| 基线 | JSON-RPC 2.0 换行分隔文本（对齐官方 Python/TS SDK 传输语义），按 id 分发响应，工具执行逻辑与 ohmcpd 完全一致（复用同一 ToolRegistry） |

## 3. 单元测试（`cargo test --workspace`，42 单元 + 15 端到端集成，共 57 项全部通过）

覆盖率（`cargo llvm-cov`，排除基准/演示测量工具本身，于 57 项测试时量得）：
**行覆盖 91.5%、函数覆盖 79.8%**；其中安全模块（认证/加解密/ACL）行覆盖 100%，
帧编解码 99.4%，软总线适配 91.0%，服务端会话处理 89.8%，共享内存通道 86.5%。

端到端集成测试（`crates/ohmcpd/tests/`）覆盖协议错误路径、安全边界与运维行为：
错误令牌拒绝、未认证请求 -32001、未知工具 -32601（且会话存活）、
CACHE_REF 命中/非幂等不入缓存、SHM 32 路并发正确性、大小混合回退、
帧内-SHM 路径一致性、上行大请求经双向 SHM 绕过套接字（128KB 参数 ×4
往返线上字节 < 8KB 且内容一致）、**前向保密加密会话端到端调用、连接数上限拒绝
（-32005 且既有会话不受影响）、优雅停机（在飞会话收尾 + socket 清理）、
resources/prompts 全链路往返（含未知资源/模板错误路径）、取消通知
（已取消请求跳过执行回 -32800 且会话存活）、进度通知（Progress 先于结果送达回调）**。

| 模块 | 用例 |
|---|---|
| ohmcp-core | 帧编解码往返、半包增量解码、非法 magic/版本/消息类型拒绝、超限帧拒绝、随机字节流不 panic、消息类型映射、RPC 结构序列化、resources/prompts 方法映射（10） |
| ohmcp-cache | 小 payload 不压缩、大 payload 压缩往返、缓存命中/未命中、TTL 过期、LRU 淘汰（5） |
| ohmcp-security | HMAC 验证/拒绝、会话密钥派生一致性、**X25519 密钥协商双方一致/每会话唯一**、加解密往返、错误密钥/篡改/截断密文拒绝、AAD 篡改拒绝、nonce 唯一性、会话前缀随机化、ACL 授权/拒绝（14） |
| ohmcp-client | 管线明文/压缩往返、加密往返、篡改 request_id 解密失败（3） |
| ohmcp-transport | 双工 100 帧批量写读往返、SHM 环往返/跨圈回绕/背压/非法引用拒绝、SCM_RIGHTS fd 传递与拒绝（7） |

## 4. 性能测试方案

`ohmcp-bench` 在同一进程内启动基线服务端与 ohmcpd（开启认证 + 加密），
用同一工具注册表消除工具执行差异，仅度量协议栈差异。指标：ops/s、
p50/p99 时延（HDRHistogram，µs）、线上字节。

| 场景 | 负载 | 考察点 |
|---|---|---|
| latency-echo | 5000 次小消息 echo，顺序 | 帧解析/信封开销、端到端时延 |
| bulk-kb-search | 5000 次不同 query 的 kb.search（~3KB 结果） | 压缩、批量数据通路 |
| repeat-cached | 5000 次调用轮转 10 个热点 query | 内容寻址缓存 + CACHE_REF |
| bulk-doc-64k | 500 次整文档拉取（~64KB 结果） | 大 payload 压缩、字节效率、共享内存通道 |
| upload-256k | 100 次 ~256KB 参数注入 echo | 请求上行大 payload、双向共享内存 |
| pipeline-64 | 单连接 64 并发 worker × 80 次 | 多路复用、队头阻塞 |
| concurrent-16 | 16 个 Agent 连接 × 500 次 | 多连接并发扩展性 |

### 大 payload 扩展性（帧内 vs 共享内存通道，非幂等 `kb.blob`）

同一负载分别经常规压缩+加密帧与共享内存通道传输，观察随大小的变化：

| 负载 | 栈 | ops/s | p99(µs) | 套接字字节/次 |
|---|---|---|---|---|
| 16 KiB | ohmcp | 20 220 | 53 | ~842 |
| 16 KiB | **ohmcp-shm** | 20 460 | 59 | **~130** |
| 64 KiB | ohmcp | 7 252 | 193 | ~2 213 |
| 64 KiB | **ohmcp-shm** | 7 969 | 152 | **~132** |
| 256 KiB | ohmcp | 1 995 | 667 | ~8 082 |
| 256 KiB | **ohmcp-shm** | 1 964 | 787 | **~139** |
| 1024 KiB | ohmcp | 347 | 4 911 | ~30 970 |
| 1024 KiB | **ohmcp-shm** | **403（+16%）** | **3 011（−39%）** | **~160** |

结论：`kb.blob` 为高冗余文本，LZ4 已大幅压缩帧内字节，故中小负载下 CPU（压缩/加密）
为瓶颈、两者吞吐接近；但共享内存通道的套接字字节恒定在 ~130 字节（12 字节引用 + 帧头），
不随负载增长，且在 1 MiB 负载下吞吐 +16%、p99 −39%——**SHM 优势随 payload 增大而扩大，
在低压缩比 / 大负载场景收益最显著**。

### 大请求上行（双向共享内存，`upload-256k`）

客户端注入 ~256KiB 参数（端侧 RAG 文档上传式负载），请求与结果两个方向
均走共享内存环，帧内仅携带 12 字节引用：

| 栈 | ops/s | p50(µs) | 套接字字节/次 |
|---|---|---|---|
| baseline | 985 | 956 | ~524 484 |
| ohmcp（帧内） | 1 036 | 911 | ~2 271 |
| **ohmcp-shm（双向）** | **2 662（+81%~+170%）** | **319（−47%~−67%）** | **~66（−99.99%）** |

（吞吐提升区间为多次完整基准运行观测范围，大请求场景基线抖动较大）

配套优化：非幂等工具不再计算缓存键（SHA-256 哈希大参数的开销只在
可缓存工具上支付），客户端对超阈值参数同样跳过缓存键哈希。

### 结论

- **通信效率**：大结果 −81%、热点重复调用 −94% 线上字节，远超赛题 ≥10% 要求；
- **时延/吞吐**：小消息、缓存命中、单连接多路复用场景全面领先（最高 +89% 吞吐、
  p99 尾延迟 763µs → 286µs）；
- **安全代价可量化**：全程 AEAD 加密下仍取得上述成绩；bulk 场景加密开销
  （每次 ~3KB 双向 AEAD）约抵消协议增益，为可按需关闭的合理折中；
- **大 payload 场景**：64KB 文档拉取常规帧内传输吞吐 +24%、线上字节 −95.6%；
  启用共享内存通道（memfd + SCM_RIGHTS，同设备零套接字拷贝）后吞吐 **+52%**、
  p99 −67%、套接字字节 **−99.8%**（帧内仅 12 字节引用）；在软总线 / BLE /
  低功耗蜂窝等带宽与能耗受限链路上，字节数直接决定时延与功耗，收益进一步放大；
- 复现：`cargo run --release -p ohmcp-bench -- --json bench-results.json`。

### 端到端场景演示

`cargo run --release -p ohmcp-bench --bin demo`：单 ohmcpd 守护进程 + 3 个并发
Agent（语音助手 / 系统调度 / 计算智能体），演示认证握手、加密信道、工具调用、
CACHE_REF 命中（重复查询 98µs → 47µs，线上仅 32 字节）、未知工具拒绝、
错误令牌拒绝，全部断言通过。

## 6. 安全测试

| 用例 | 方法 | 结果 |
|---|---|---|
| 令牌不过网 | 认证帧仅含 HMAC(token, nonce) | ✅ 通过（协议构造保证 + 单测） |
| 前向保密 | X25519 临时密钥交换，会话密钥掺入 ECDH 共享秘密，临时私钥握手后销毁 | ✅ 单测 + e2e（令牌事后泄露不可解历史流量） |
| 连接洪泛（DoS 面） | 超出 `--max-conns` 上限的连接收 -32005 后关闭 | ✅ e2e（既有会话不受影响，释放后可再接入） |
| 供应链 | cargo-deny（advisories/bans/licenses/sources）接入 CI | ✅ 全部通过，持续门禁 |
| 错误令牌拒绝 | `verify_response` 常数时间比较 | ✅ 单测通过 |
| 未认证访问拒绝 | 未 Auth 直接 CallTool | ✅ 返回 -32001 |
| 密文篡改检测 | 修改 request_id / 密文后解密 | ✅ AEAD 校验失败（单测） |
| 越权工具调用 | ACL 未授权工具 | ✅ 返回 -32002（单测覆盖 ACL） |

## 7. 适用范围声明与局限

**基线适用范围（如实声明）**：本报告中的“基线”是对齐官方 Python/TS MCP SDK
传输语义（JSON-RPC 2.0 换行分隔）的自实现，而非官方 SDK 本体；数据均来自
同机 UDS 微基准（Linux x86_64），非真机 OpenHarmony + 真实 DSoftBus 链路。
两端复用同一 ToolRegistry 以消除工具执行差异，数字反映的是协议栈层面的
相对改善，真机/跨设备绝对数值会随硬件与链路变化。

其他局限与后续工作：

- bulk 大结果 + 加密场景吞吐与基线持平：可引入硬件加速 AEAD 或按连接协商加密；
- MCP 语义已覆盖 tools/resources/prompts + 取消/进度通知；订阅（subscriptions）待补；
- 与 OpenHarmony 分布式软总线的跨设备工具调用集成（适配层 PoC 已落地，待真机验证）。
